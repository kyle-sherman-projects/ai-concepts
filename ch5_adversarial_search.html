Chapter Core Idea

In competitive multi-agent environments, agents have conflicting goals. Instead of a fixed environment, you now face an active opponent. The key shift: your solution can't be a fixed action sequence — it must be a conditional strategy accounting for every possible opponent response.

Three stances toward adversarial agents: treat as an aggregate economy · treat as a nondeterministic environment · model explicitly with game-tree search ← this chapter

Types of Games Covered

MAX  Deterministic Perfect-Info

Chess, Go, Checkers, Tic-Tac-Toe
Two-player, turn-taking, zero-sum. Most thoroughly studied class. Minimax and Alpha-Beta apply directly.

CHANCE  Stochastic Games

Backgammon, Monopoly
Dice or other random elements added. Requires expectiminimax — takes weighted average over chance outcomes.

PARTIAL  Imperfect Information

Poker, Bridge, Kriegspiel
Players can't see the full state. Requires belief states and can call for randomized strategies (equilibria).
Zero-sum clarification: Chess payoffs sum to +1 per game (win=1, loss=0, draw=0.5 each), not zero — but it's called zero-sum because one player's gain exactly equals the other's loss. The relative payoffs sum to zero.

Minimax Formula

MINIMAX(s) =
  UTILITY(s, MAX)                    if IS-TERMINAL(s)
  maxa ∈ ACTIONS(s) MINIMAX(RESULT(s,a))  if TO-MOVE(s) = MAX
  mina ∈ ACTIONS(s) MINIMAX(RESULT(s,a))  if TO-MOVE(s) = MIN
Key properties: Complete depth-first search · Time: O(bm) · Space: O(bm) or O(m) if actions generated one at a time · Not feasible for chess (1040 nodes) but defines what optimal means.

MAX vs MIN Strategy

MAX represents our agent — wants to maximize utility. Assumes opponent plays optimally.
MIN represents the opponent — wants to minimize MAX's utility. Minimax assumes MIN is also optimal.

MAX's strategy must be contingent — for every possible MIN move, MAX must have a response. This is why the solution is a conditional plan, not a sequence.

Multiplayer Extension

With more than 2 players, each node's value becomes a vector — one utility value per player. Each player picks the action that maximizes their own component.

Alliance formation: Multiplayer games often produce informal alliances — coalitions emerge naturally as agents try to maximize individual utility. The game-tree framework can model this by examining each player's vector payoff.

Alpha–Beta Key Insight

Core principle: If Player has a better choice elsewhere in the tree, Player will never move to node n. Once we've seen enough of n's descendants to confirm this, we can prune the remaining descendants without evaluating them. The result is identical to minimax — but far fewer nodes examined.
α = best (highest) value found so far for MAX. Think: "at least"
β = best (lowest) value found so far for MIN. Think: "at most"

Prune when: current node value ≤ α (at MIN node) or ≥ β (at MAX node)

Move Ordering Impact

The effectiveness of pruning depends entirely on which nodes are examined first.

Best ordering (perfect)O(bm/2)
Dynamic ordering (practical)~O(b3m/4)
Random orderingO(bm)

With best ordering, alpha–beta can explore trees twice as deep as minimax in the same time. Killer move heuristic + iterative deepening gets close to this.

Practical Enhancements

KILLER MOVE HEURISTIC

Try moves that caused cutoffs in sibling nodes first. Prior "killer moves" are reused across similar positions.

ITERATIVE DEEPENING

Run alpha–beta to increasing depths. Results from shallower searches inform move ordering for deeper ones.

H-Minimax — Cutting Off the Search

H-MINIMAX(s, d) =
  EVAL(s, MAX)                        if IS-CUTOFF(s, d)
  maxa∈ACTIONS(s) H-MINIMAX(RESULT(s,a), d+1)  if TO-MOVE(s)=MAX
  mina∈ACTIONS(s) H-MINIMAX(RESULT(s,a), d+1)  if TO-MOVE(s)=MIN
Replace UTILITY (exact terminal value) with EVAL (heuristic estimate). Replace IS-TERMINAL with IS-CUTOFF (depth limit or other criteria). This makes real games tractable.

Weighted Linear Evaluation Function

EVAL(s) = Σ wi · fi(s)

= w1f1(s) + w2f2(s) + … + wnfn(s)

Each fi is a feature (e.g., number of queens, rooks, pawns). Each wi is its weight (e.g., queen=9, rook=5, pawn=1).

Independence assumption: Weighted linear evaluation assumes feature contributions are independent. In reality they're not (a bishop's value changes depending on pawn structure). Real programs use nonlinear combinations.

Requirement: UTILITY(loss) ≤ EVAL(s) ≤ UTILITY(win). Doesn't need to be linearly correlated with win probability — just correctly ordered.

Search Issues & Mitigations

HORIZON EFFECT

Unavoidable damage lurks just beyond the search depth. The agent uses delaying tactics to "push" the problem past the horizon — appearing to avoid it while really just deferring.

SINGULAR EXTENSIONS

When one move is clearly better than all others, extend search beyond the normal cutoff for that line. Mitigates horizon effect in forced-move positions.

QUIESCENCE SEARCH

Only apply EVAL to "quiet" positions — no captures, checks, or tactical threats pending. Avoids misvaluing positions mid-tactic.

FORWARD PRUNING

Beam search or PROBCUT: consider only the N best moves rather than all legal moves. Faster but risks pruning the true best move. Shannon's "Type B" strategy.

Opening & Endgame Table Lookup

Many programs replace search entirely for known positions:

Opening books: Databases of master-level opening sequences. Statistics from millions of played games identify which openings most often win. The computer looks up moves rather than searching.
Endgame tablebases: Computer analysis solves endgames completely via retrograde minimax. A policy maps every possible state to the optimal move — this goes far beyond human ability in 7-piece endgames.

Monte Carlo Tree Search — Four Phases

MCTS avoids heuristic evaluation functions entirely. Instead, it estimates state values by running many random playouts to terminal states and averaging results.

01
SELECTION
Walk down the tree using the selection policy (UCT) until a leaf node is reached.
02
EXPANSION
Add a new child of the selected leaf node to the search tree.
03
SIMULATION
Run a playout (rollout) from the new node to a terminal state using the playout policy. Not recorded in tree.
04
BACK-PROP
Propagate the playout result back up through all ancestor nodes, updating N(n) and U(n).
Selection: Starting at the root, repeatedly choose a child according to the UCT selection policy. This balances exploring new moves against exploiting moves that have done well. Continue until reaching a node with unexpanded children.

UCT — Upper Confidence Bounds Applied to Trees

Click each term to understand it:

UCB1(n) =   U(n) / N(n)   +   C   ×   √( log N(PARENT(n)) / N(n) )
Click a term above to see what it means.
C constant: Theoretically √2, but practitioners try multiple values and pick what performs best. C controls the exploitation/exploration tradeoff — higher C = more exploration.
Efficiency: Playout time is linear in tree depth (only one move chosen per level), not exponential — enabling thousands of simulations per second. No heuristic evaluation needed; just the rules of the game.

Why MCTS for Go?

Alpha-beta fails at Go: Branching factor starts at 361. Alpha-beta limited to only 4–5 ply. Defining a meaningful evaluation function is extremely hard — material advantage barely predicts outcomes.

MCTS handles both problems: no evaluation function needed, and branching factor is handled statistically rather than exhaustively.

MCTS Limitations

Tactical blindness: The stochastic nature means a critical "one move changes everything" line might never be simulated. Type-B pruning: vital lines can be completely missed.

Modern programs (AlphaGo, AlphaZero) combine MCTS with deep neural networks for both the selection policy (replacing UCT's uniformity) and a value network (replacing random playouts).

Expectiminimax — Adding Chance Nodes

EXPECTIMINIMAX(s) =
  UTILITY(s,MAX)                             if IS-TERMINAL(s)
  maxa EXPECTIMINIMAX(RESULT(s,a))          if TO-MOVE(s) = MAX
  mina EXPECTIMINIMAX(RESULT(s,a))          if TO-MOVE(s) = MIN
  Σr P(r) · EXPECTIMINIMAX(RESULT(s,r))  if TO-MOVE(s) = CHANCE
CHANCE nodes compute the expected value: sum of all child values weighted by their probability. Terminal and MAX/MIN nodes work exactly as before.
Critical constraint: The evaluation function must return a positive linear transformation of win probability (not just ordinal rankings). Monotone transformations that preserve order can give wrong answers in stochastic trees.

Why Alpha-Beta Struggles in Stochastic Games

Alpha-beta's power comes from ignoring branches that won't happen given best play. But with dice rolls before every move:

Even the most likely move occurs only ~1/2 the time (the dice must come out right first). There are no "likely sequences." Possibilities multiply enormously. Detailed plans become pointless because the world probably won't cooperate.

Partially Observable Games

When players can't see the full board, uncertainty arises from opponent choices, not just randomness.

Belief states: The set of all logically possible board states given the full percept history. AND-OR search over belief-state space finds guaranteed checkmates.
Probabilistic checkmate: A strategy that works for some belief states may succeed accidentally. Even a lone king can be found with probability 1 through random movement — detection is guaranteed eventually.

Optimal play in partially observable games requires randomized strategies — predictable play gives information to the opponent. Game-theoretic equilibria specify optimal randomized strategies for each player.

Four Limitations (R&N's Framing)

1. Heuristic errors — Alpha-beta is vulnerable to flawed evaluation functions. A wrong EVAL cascades through backed-up values.
2. Wasted computation — Both algorithms compute move values even when one move is obviously best (e.g., only one legal move). Metareasoning could skip this.
3. Move-level reasoning — Humans think at higher abstraction (e.g., "trap the queen"). Algorithms think move-by-move. Planning (Ch. 11) addresses this.
4. Hand-crafted evaluation — Early programs relied on human expertise for features/weights. Modern programs use machine learning (self-play, neural networks) to learn these automatically.

When to Use Which

Use Alpha-Beta when: branching factor is manageable, a good evaluation function can be defined, game is deterministic and perfect-information.
Use MCTS when: branching factor is very high (Go), evaluation function is hard to define, or the game is fully simulatable to terminal states quickly.
Use Expectiminimax when: game includes dice or other chance elements (Backgammon). Must ensure evaluation function is a positive linear transform of win probability.
Combine: AlphaZero uses MCTS + neural network policy (selection) + neural network value (replacing rollouts). Both algorithms benefit from iterative deepening and opening/endgame tables.
